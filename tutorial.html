Congratulations! You are now the proud owner of a Gregory Loden Run-class autoclicker!

 This wonderful piece of software allows you to write your own mouse-and-keyboard-controlling programs using text files and a simple syntax, allowing for easy and quick program creation.
 To start off, there is a folder named "scripts" in the same directory as the run.class file. This is where all autoclick scripts will be located. To be recognized, all scripts must end with .txt. Two sample utility scripts have been provided, in addition to a variety of example scripts.
 To run a script, type java run ??? ... in a command line/console/terminal (for the purpose of this tutorial, we'll use "command line" for input and "console" for output). This will run the script named ???. You can run scripts in subfolders by including slashes, such as java run example/001. The ... is an arbitrary amount of parameters to be passed to the autoclicker for use in the script. We'll see why we need these later.
 In all commands, command calls and parameters are separated by whitespace tabs or spaces. Tabs and spaces are recognized by the program as separators.

SECTION 1- Standard Autoclicking Features

 The core of any autoclicker is the ability to perform right clicks, left clicks, mouse moving, and pausing.
 In this autoclicker, a command looks like this: 
	left	[x]	[y]
 When the autoclicker's line pointer reaches a line, the command on that line is executed
 In this case, a left click. The name goes at the beginning, followed by the parameters. In this tutorial file, anything, within [] brackets stands for something that would get replaced by actual values written by the scriptmaker (you or someone else).
 The snippet above is for a left click; let's take a look at how it works. When the autoclicker reaches a line with this left command, the mouse moves to the x-[x] and y-[y] coordinates provided, and left clicks that spot. For example, 
	left	100	200
will left-click at x-100, y-200.
 All commands take up one line only, and contain only lowercase letters. Uppercase letters are not used by any words defined by the autoclicker. You can also leave yourself comments by starting a line with anything except one of the commands. All comments get removed upon running the script, as if they were never there. Here are examples of two more commands: 
	// Right click:
	right	[x]	[y]
	
	~ Middle click:
	middle	[x]	[y]
	
	. Move:
	move	[x]	[y]
 right will move and perform a right click, middle will move and perform a middle click, and move will move the mouse without clicking. Any number of tabs and spaces can be added between parameters and commands, and the interpreter will pass right over them. That means that 
	move	[x]	[y]
is the same as 
	   move			[x] [y]
but not 
	move	[x][y]
as whitespace is needed to separate parameters (so 53 4 is different than 534). In commands, when writing a number, only the digits (and possibly negative sign before the digits) are taken into account, so these two commands, 
	move	1000	234
	move	1,000	2jrgfsd3j.4
will perform the same action (move the mouse to x-1000, y-234).
 And because it covers all your core autoclicking needs, Run of course supports pausing for certain periods of time. The command 
	pause	[t]
will pause for [t] milliseconds. For instance, 
	pause	1000
will have the autoclicker pause for 1 second before resuming to the next line.

 With all of these commands, we can now start to form autoclick scripts. Take a look at the example/001 script, a simple example containing clicks, moves, and pauses.

 Each command can have multiple different forms that accept different amounts of parameters; the contract will list the syntax of each form and what it does. Some forms will be explained further in the tutorial than when the contract is presented.
 Let's take a look at the different forms in both the left and right commands: 
	left	[num]
	right	[num]
 Rather than move and click, this will have the mouse will perform differently depending on what [num] is: 2 will press and release the mouse, anything less than 2 will press the mouse, and anything greater than 2 will release the mouse. Neither of these two forms will cause the mouse to move. Take a look at this: 
	move	100	100
	pause	1000
	left	1
	pause	1000
	move	150	200
	pause	1000
	left	3
 This would move the mouse to x-100, y-100, wait a second, press the left mouse button, wait a second, drag the mouse to x-150, y-200, wait a second, and release the button. This snippet is provided as example/002.
 In addition to moving the mouse position, you can also move the mouse scroll wheel with the following commands: 
	scroll	up
	scroll	down
	scroll	up	[number]
	scroll	down	[number]
 The scroll command will execute one scroll "click" in the given direction, or [number] "clicks" if a number is provided.

scroll:

scroll	[direction]	[amount?]

	input:
		[direction] - either "up" or "down"
		[amount] (optional) - number of "clicks" to scroll
	output:
		- scrolls 1 or [amount] "clicks" in the provided direction

 The second form of pause has been explained, but let's take a look at the first and third forms. You can have the pause command by itself: 
	pause
 This will pause until you press ENTER on the command line that you ran the script from.
 The remaining forms of pause all deal with screen colors: 
	pause	[x]	[y]	[r]	[g]	[b]
 will pause until the red-green-blue color at x-[x], y-[y] is red-[r], green-[g], and blue-[b]. This is useful for verifying that a particular action causes the screen to change in the right way.
 This five-parameter pause can also take a sixth parameter, combining both ways of pausing. This number, a time in milliseconds, tells the pause how long to wait before ignoring the color and resuming the program. This is useful if the color does not change in time, but the program can catch this and handle it. Whichever comes first, the pause will resume. This command, 
	pause	123	456	255	255	255	1000
will wait for the pixel to change, but if a second goes by, it will stop pausing and resume anyway.
 You may come across a need to end your script early. For this purpose, moving the mouse during pauses (except for the no-parameter pause) will stop the script and end the program (you can change this, but we'll get to that later).
 You can also have pauses that wait until a specific pixel is NOT a specific color: 
	pausenot	[x]	[y]	[r]	[g]	[b]
 This will pause until the color is in the range of colors that pause will pause on.

 And of course, we can't forget looping! For this, the following command is provided: 
	jump	[LABEL]
 This will move the line pointer to a label in the format 
	:[LABEL]
 For example, 
	:Endless
	jump	Endless
is an infinite loop.

jump:

jump	[label]

	input:
		[label] - the name of a label to jump to
	output:
		- moves the script pointer to the provided label

(labels):

:[label]

	output:
		- provides a label for the jump and call commands

 Labels must contain no whitespace. Here's an example of clicking in a loop: 
	:main
	right		100	200
	pause		110	210	255	255	255
	left		110	210
	pausenot	110	210	255	255	255
	move		90	190
	pause		500
	jump		main
 This will right click at x-100, y-200, wait until the pixel at x-110, y-210 is white (r-255, g-255, b-255), left click at x-110, y-210, wait for it to stop being white, move the mouse to x-90, y-190, wait half a second, and repeat. You can find another example script with jumping in example/003.
 Run will not perform any automatic looping. If the jump were removed, the above script would end after the last pause.

 In addition to standard jumping, there is also a command to use a part of the script in multiple places at once! This is done with two related commands. The first one, 
	call	[label]
does the exact same thing that jump does, but it will remember where it got called. Paired with it is another command, 
	jumpback
which will return to the most recent call command that got executed. That means that if there are multiple calls, jumpback commands will go back down the list of call commands in the reverse order that they were called.

call:

call	[label]

	input:
		[label] - the name of a label to jump to
	output:
		- moves the script pointer to the provided label, remembering the current position

jumpback:

jumpback

	output:
		- returns to the position of the most recently performed call command

 Here's an example of multiple call and jumpback commands: 
	call	test
	call	test
	call	test
	jump	end
	
	:test
	left	100	100
	call	test2
	pause	1000
	jumpback
	
	:test2
	pause	500
	move	200	100
	jumpback
	
	:end
 This will left click at x-100, y-100, pause for a second, and repeat that twice more. You can find this script as example/004.

 The autoclicker can also run scripts from within a script: 
	run	[script]	...
will run the script [script] as if you typed 
	java run [scripts] ...
on the command line with the parameters ... like a regular command line run execution. Scripts cannot have spaces in the name.

run:

run	[script]	[input...?]

	input:
		[script] - the name of a script
		[input] (optional) - 0 or more command line inputs to pass to the script
	output:
		- runs the provided script with any provided inputs as if run from the command line

SECTION 2- Programming Language Features- Variables

 To improve what you can do with it, Run serves as a very simple programming language! This of course means that you'll need some variables and a means of changing them. You can define variables with this command: 
	create	[type]	[name]	[value]
 The above command will create a variable of type [type] called [name] with starting value [value]. These variables will probably be of type int (integer), used to base clicks on other data, but there are multiple other variable types. All variables are for use only within the active script. Scripts cannot use variables from scripts they called, nor from the scripts that called them. Variables are case-sensitive.
 Here's a bunch of variable creation examples, using all of the currently available variable types: 
	create	int	x	100
	create	boolean	Good?	true
	create	color	white	255	255	255
	create	color	brown	at	123	456
	create	timer	time	0
	create	screenshot	pic	100	55	80	60
	create	int	y	x
	create	color	brown~2	brown
	create	screenshot	pic2	pic
 Let's go over these examples!

 In the first example, the variable x was created as an int, or a whole number. To create an integer, the only parameter that must be supplied is the value to store to it.
 In the second example, Good? is created as a boolean. A boolean value simply means either a true or a false. To create a boolean, the only value that must be provided is a true or a false. Booleans serve a special purpose which will be mentioned later on.
 In the third example, white was created as a color, a type used for special commands that track colors. This is one of two ways to initiate a color, both of which take three parameters. The first way, used to create white, is to give the red, green, and blue values of a color, in that order.
 In the fourth example, brown demonstrates the second way to create a color: the parameters are the word at, followed by two more parameters- the x and y coordinates of a pixel location. The value stored to the variable is the color at that pixel.
 In the fifth example, the variable time was created as a timer. This timer's value is an integer, and increases by 1 every millisecond. To create a timer, the only value that needs to be provided is its initial value.
 In the sixth example, the variable pic was created as a screenshot. To create a screenshot, you supply the top and left coordinates of the image on the screen to capture, followed by the width and the height. pic is the screenshot at x-100, y-55 with a width of 80 and a height of 60. These will be used in special screen-processing explained later in the tutorial.
 In the seventh, eigth, and ninth examples, variables were created using other variables as a single parameter. This will create a variable with the same value of the other variable (or the same screenshot image and top left corner). All variable types can be made by providing another variable of the same type (booleans, ints, and timers can all accept each other as inputs, more on that later).

 Your variable names can consist of any non-whitespace characters, but cannot start with a period. Be careful with your naming conventions to avoid errors in your scripts related to variable overwriting. Some names are reserved by the autoclicker.

create:

create	[type]	[name]	[value]

	input:
		[type] - data type of the variable, either "int", "boolean", or "timer"
		[name] - variable name
		[value] - a value to set to the variable
	output:
		- initializes a variable to the specified type and value


create	color	[name]	[r]	[g]	[b]

	input:
		[name] - variable name
		[r] - the red value to store to the color
		[g] - the green value to store to the color
		[b] - the blue value to store to the color
	output:
		- initializes a color with the 3 given values


create	color	[name]	at	[x]	[y]

	input:
		[name] - variable name
		[x] - the x coordinate of a pixel to take the color of
		[y] - the y coordinate of a pixel to take the color of
	output:
		- initializes a color with the color of the provided pixel


create	color	[name]	[color]

	input:
		[name] - variable name
		[color] - a color variable
	output:
		- initializes a color with the values of the other color


create	screenshot	[name]	[x]	[y]	[w]	[h]

	input:
		[name] - variable name
		[x] - the left x coordinate of a screenshot to capture
		[y] - the top y coordinate of a screenshot to capture
		[w] - the width of a screenshot to capture
		[h] - the height of of a screenshot to capture
	output:
		- initializes a screenshot with a screen capture of the given area of the screen


create	screenshot	[name]	[screenshot]

	input:
		[name] - variable name
		[screenshot] - a screenshot variable
	output:
		- initializes a screenshot with the image and top left corner of the other screenshot variable

 So now that we have variables, we can use them in other commands! Using the above variable assignments from the examples, the command 
	left	x	y
would left click at x-100, y-100.
 In addition to copying variables, they can also be used as regular parameters. This command, 
	create	color	brown	x	y	0
would create a color with a red value of 100, green value of 100, and blue value of 0.
 color variables are used to replace sets of 3 parameters that make up a color. Assuming the same variable assignments, these two commands, 
	pause	123	456	255	255	255
	pause	123	456	white
would function exactly the same, pausing until the pixel is white. This is the fourth form of pause and the second form of pausenot. The extra parameter at the end will still function, so 
	pause	123	456	white	1000
will quit out after a second. Of all commands mentioned so far, only pauses and pausenots use colors, but they are used in commands later in the tutorial.

 In addition to creating variables, variables already created can be set to new values like this: 
	set	[name]	[value...]
 Setting variables works just like creating variables, for all types of variables, except you don't need a variable type. set changes the variable of name [name] to the value that the parameters [value...] produce. Variables can also technically be changed with create, and you can change the type of the variable by doing this, but set is recommended for just changing the value.
 Here are some examples of set: 
	set	x	200
	set	time	43
	set	brown	at	43	96
 Still assuming the same variable assignments, this would change x's value to be 200. So now, 
	left	x	y
would left click at x-200, y-100. All variables are ints unless they have been created as something else via the create command. This means that 
	set	z	100
would set z to 100 even though z has not been created. You can check out some examples of create, set, and using variables in commands in example/005 and example/006.

	input:

 There is also a special set command for using command line parameters with the script: 
	setargto	[prefix]	[name]
 This works as if you called 
	set	[name]	[value]
assuming that [prefix][value] is one parameter to the program, and is a single string of characters without spaces or tabs.
 As an example, say you had a script called test and you typed this on the command line: 
	java run test abc123
 If you had 
	create	int	doremi	0
	setargto	abc	doremi
within that script, it would end up changing doremi as if you called 
	set	doremi	123
which sets the variable to 123, omitting the abc in the parameter. The provided script example/007 will move the mouse to the coordinates given on the command line. Try it out by running something like java run example/007 x200 y100.
 Now, since some of the variable creations take in multiple parameters, but input parameters cannot include tabs or spaces, you will have to type commas between them when typing them on the command line. This means that parameters should be typed like alphaat,123,456 and beta255,255,255 for the following commands: 
	create	color	test1	0	0	0
	create	color	test2	0	0	0
	setargto	alpha	test1
	setargto	beta	test2
 These will end up acting as if you had these commands instead: 
	create	color	test1	at	123	456
	create	color	test2	255	255	255
 If you call setargto and there was no parameter that started with [prefix], the command will do nothing.

setargto:

setargto	[prefix]	[name]

	input:
		[prefix] - text that will appear before parameters in the command line
		[name] - the name of the variable to set
	output:
		- sets the variable to the values specified on the command line



 In addition to using set to store to variables, there are 5 available integer arithmetic-and-assignment commands: 
	+=	[name]	[value]
	-=	[name]	[value]
	*=	[name]	[value]
	/=	[name]	[value]
	%=	[name]	[value]
 The first four will respectively add, subtract, multiply, and divide the value of [name] by [value], and store the result back in [name]. The fifth one will set name to the remainder of [name] divided by [value] (modulus division). Here's an example: 
	create	y	1
	+=	y	1
	*=	y	9
	%=	y	4
	-=	y	10
	/=	y	-8
 This will create a variable y equal to 1, add 1 to it to give it a value of 2, multiply it by 9 to give it a value of 18, modulo-divide it by 4 to give it a value of 2, subtract 10 to set it to -8, and divide it by -8 to set it back to 1. All operators work like most programming languages with 32-bit integers: division will result in an integer rounded down, overflows will make the number go crazy. The example/008 script uses these commands.

+=	[name]	[value]

	input:
		[name] - variable name
		[value] - integer value
	output:
		- increases the value of the variable by [value]

-=:

-=	[name]	[value]

	input:
		[name] - variable name
		[value] - integer value
	output:
		- decreases the value of the variable by [value]

*=:

*=	[name]	[value]

	input:
		[name] - variable name
		[value] - integer value
	output:
		- multiplies the value of the variable by [value]

/=:

/=	[name]	[value]

	input:
		[name] - variable name
		[value] - integer value
	output:
		- divides the value of the variable by [value], rounding to 0

%=:

%=	[name]	[value]

	input:
		[name] - variable name
		[value] - integer value
	output:
		- sets the value of the variable to the remainder of [name] / [value]



 Run also includes some useful special system variables! The variables 
	.mousex
	.mousey
represent the mouse's current x- and y-coordinates. So for example, 
	create	int	clickx	.mousex
	create	int	clicky	.mousey

SECTION 3- Programming Language Features- Conditionals and Loops

 So now we have autoclicking and variables. What more could we possibly want? Conditionals and for loops, that's what!
 Earlier, it was mentioned that you could create variables of type boolean. The main use of these is for the if commands that look like this: 
	if	[boolean]
		[command1]
	[command2]
 The way this works is that if [boolean] is true, both [command1] and [command2] get called. If [boolean] is false, [command1] doesn't get called, but [command2] does.
 In addition to this format, if has another syntax that can be used, and it looks like this: 
	if	[boolean]
	[
		[command1]
		[command2]
		...
	]
	[some-other-commands]
 Right after the if command, a line with a [ bracket at the beginning signifies the start of the commands to be run conditionally. If [boolean] is false, the pointer jumps to the ] bracket and continues. You can also use ( parentheses and { braces, or mix and match them, and they will all work the same.
 Now, if commands are very line-sensitive, so make sure your line formats are correct (although comments can be placed anywhere). Take this for example: 
	create	boolean	test	[value]
	if	test
		:here
		[command]
 Since if only skips the next line, the label will get skipped instead of [command]. So whether [value] is true or false, [command] will get executed anyways. Brackets are also sensitive: a bracket must appear immediately after the if command, or it won't get recognized. The following, 
	if	false
	:here
	[
		[command]
		...
	]
will execute [command] since the brackets are not linked to the if command. In addition, all brackets link to each other, so make sure they're all paired correctly.
 Note that the following,
	if	false
	//abc
	[
		[command]
		...
	]
will not execute [command] since the comment is removed and the bracket is paired with the if command.

if:

if	[value]

	input:
		[value] - result value
	output:
		- skips the following line or block if [value] evaluates to true

 So now that we have our if command, we can use it with jumps for more complex conditionals: 
	create	boolean	sety	true
	create	int	x	100
	create	int	y	100
	if	sety
		jump	thisisalabel
	set	x	150
	jump	end
	:thisisalabel
	set	y	150
	:end
 This will set y to 150 since the value given to it is true. If the true were replaced with false, x would get set to 150. The above is also equivalent to the following: 
	create	boolean	sety	true
	create	int	x	100
	create	int	y	100
	if	sety
	[
		set	y	150
		jump	end
	]
	set	x	150
	:end
 An example of these if forms is available in example/011.

 Constant and variable booleans are not the only booleans that can be put in an if clause. Clauses can contain multiple operations, and if commands are not the only places that these clauses can be used! These clauses are big enough that they get their own section, and we haven't even gotten to for loops!

SECTION 3.1- Clauses

 A clause is a big paramater consisting of a bunch of operators and variables. For many commands, a single value can be replaced with one of these clauses (a full list can be found in the README file). The at operator used in colors is only available in create and set commands, and only when placed immediately after the variable name.

 Let's go over how the variable types are stored and represented in clauses. A clause stores all types as integers. The variable types int, boolean, and timer can each be represented with a single integer; other variable types are used with other operators to obtain integer representations.
- ints are stored as 32-bit numbers and are simply represented as themselves.
- booleans are stored as true or false. They represent 1 if they are true, and 0 if they are false. Now, in addition to a boolean being treated as a number, numbers can be used as booleans. When a result in a clause gets interpreted as a boolean, any number besides 0 is interpreted as true (0 is false); this is how if commands determine whether to skip the line/block or not. For the most part, you won't be using boolean variables as integers.
- timers are stored similarly to ints, and for the most part are treated as integers. The difference is that timers change over time.
- colors are stored as 3 integers, this is how many commands are able to use both 3 RGB values and color variables. Some operators and commands specifically require color variables.

 A clause always returns an int, but can be interpreted as a boolean when storing to a boolean variable or as the result of an if condition.

 So now that we've gotten that over with, let's go over the clauses! Here are a few examples: 
	if	[clause]
	create	[type]	[name]	[clause]
	left	[xclause]	[yclause]
	pause	[timeclause]
	setargto	[prefix]	[name]
 Note that for setargto, the clause has to be a paramater to the program, separated with commas rather than spaces. There'll be a bit on that a little later.
 Clauses can be as simple or complicated as you make them. Take a look at this: 
	if	true
 This will always pass, as the clause represents true, and nothing else. Here's another clause that does the same thing: 
	create	boolean	test	true
	if	test
 The clause is true if the boolean is true. Here's a third way to do the same thing: 
	if	23
 Since numbers can be treated as booleans, any number besides 0 represents true.
 There are six integer-value evaluation clauses: 
	if	>=	[value1]	[value2]
	if	>	[value1]	[value2]
	if	<=	[value1]	[value2]
	if	<	[value1]	[value2]
	if	==	[value1]	[value2]
	if	!=	[value1]	[value2]
 These are pretty straight-forward, they evaluate to true if [value1] is greater-than-or-equal-to, greater than, less-than-or-equal-to, less than, equal to, or not equal to [value2], respectively. Remember that booleans are represented as a 1 or a 0, so you can use them like integers here. Script example/012 contains examples of comparisons.

 Along with these basic comparison clauses for integers, Run also supports standard boolean logic with the following three commands: 
	if	not	[clause]
	if	or	[clause1]	[clause2]
	if	and	[clause1]	[clause2]
 Let's go over how these work. The not clause, 
	if	not	[clause]
will evaluate to false if [clause] IS true. For example, 
	create	int	five	4
	create	boolean	correct	false
	if	not	correct
		set	five	5
will end up setting the value of five to 5, since correct is NOT true. The or clause, 
	if	or	[clause1]	[clause2]
will end up evaluating to true if [clause1] is true, [clause2] is true, or both are. Say you wanted to run a command if either of two booleans were true: 
	create	boolean	a	true
	create	boolean	b	true
	if	or	a	b
		move	100	100
 If either a or b is true, the mouse will move to x-100, y-100. Similarly, the and clause, 
	if	and	[clause1]	[clause2]
will evaluate to true only if both [clause1] and [clause2] are true.
 Note that since integer values can evaluate to booleans, you can have clauses like 
	create	int	a	32
	if	and	a	false
		move	100	100
or 
	create	int	two	2
	if	not	a
		move	100	100
 Both of those will skip the following move command. Script example/013 contains examples of boolean operators.

 So now that we have boolean logic and arithmetic, let's combine them to form more complex expressions! Say you created these variables, 
	create	boolean	A	true
	create	boolean	B	false
	create	int	C	1
	create	int	D	7
and then you wanted to implement this expression: 
	(A and B or (C + 6 != D / 3))
 In order to use this as a clause, it has to be formatted properly. First make sure all operators are grouped with only two values: 
	((A and B) or ((C + 6) != (D / 3)))
 Next, move all operators to the left side of the parentheses, since this is the format for operators: 
	(or (and A B) (!= (+ C 6) (/ D 3)))
 After removing the parentheses, we can put it in a clause: 
	if	or	and	A	B	!=	+	C	6	/	D	3
 As soon as this line gets reached, Run starts evaluating. How this works is that the pointer starts from the right, and goes left along the expression, evaluating the operators as it goes. The variables are given their values. As soon as the pointer reaches an operator, it uses it in conjunction with the values that follow it. For instance, the pointer in this example starts at the 3, goes to the D and stores it as 6, and when it reaches the /, it stores the value of D / 3. Take a look at the sequence of evaluation steps that take place: 
												V
	if	or	and	A	B	!=	+	C	6	/	D	3
	
											V
	if	or	and	A	B	!=	+	C	6	/	7	3
	
										V
	if	or	and	A	B	!=	+	C	6	2
	
									V
	if	or	and	A	B	!=	+	C	6	2
	
								V
	if	or	and	A	B	!=	+	1	6	2
	
							V
	if	or	and	A	B	!=	7			2
	
						V
	if	or	and	A	B	true
	
					V
	if	or	and	A	false	true
	
				V
	if	or	and	true	false	true
	
			V
	if	or	false			true
	
		V
	if	true
 At the end, the result is true, so the if does not skip lines. And that's how clauses work! As mentioned before, the README contains a list of all places where values can be clauses. The above clause and other long clauses are included in example/015.
 For setgargto, the clauses must take place on the command line. Take a look at this line: 
	set	twelve	-	*	7	2	2
 To have setargto perform the same action, have something this line in your script: 
	setargto	t	twelve
 Then, when you run your script, type 
	java myscript t-,*,7,2,2
and the setargto command will perform just like the set command. Note that you cannot access variables from a setargto- variable access is limited to the script that the variables exist in, so something like this, 
	java myscript t-,*,D,2,2
would not have the D turn into 7, even though there is a variable of name D.

 There are two clause forms that check for screen pixel colors. The first one, 
	if	colorat	[x]	[y]	[r]	[g]	[b]
evaluates to true if the pixel at x-[x], y-[y] is r-[r], g-[g], b-[b], similar to the pause command. Likewise, it also has a format

	if	scan	[x]	[y]	[w]	[h]	[r]	[g]	[b]
takes a screenshot of the rectangle with top-left corner at x-[x], y-[y], with width [w] and height [h]. If any of the pixels in that rectangle are r-[r], g-[g], b-[b], then it returns true. A single scan runs about as fast as a single colorat search. Like colorats, scan also has the form 
	if	scan	[x]	[y]	[w]	[h]	[color]
which uses the color variables. Scanning also introduces two new system variables, 
	.scanx

which are the x- and y-coordinates of the pixel that triggered the scan to be true. If the scan returns false, they are both -1.
 Color searches don't have to search only on the current screen- you can also scan stored screenshot variables. The scan operator can exchange [x], [y], [w] and [h] for a screenshot like in the following:
	if	scan	[screenshot]	[r]	[g]	[b]

	create	screenshot	p	100	300	50	30
	if	scan	p	255	255	255
		move	100	100
 p stores a capture of the screen for later. When the script reaches the if command, it uses p instead of taking a new screenshot. You can also substitute [r], [g], [b] with [color] just like pauses.

	if	imagecolorat	[screenshot]	[x]	[y]	[r]	[g]	[b]

	if	imagecolorat	p	100	300	255	255	255

	create	color	pixel	imagecolor	[screenshot]	[x]	[y]
	create	color	pixel	imagecolor	p	100	300

SECTION 3.2- For Loops

 Now on to for loops! They don't really need their own section, but hey, why not? for loops run their contents once "for" each value between the starting and ending values. A for loop starts with the command 
	for	[name]	[init]	[max]	[incr]
and ends with the command 
	end
 The for command generates a variable named [name] with a value of [init]. When the script reaches the end command, it increases [name] by [incr]. If the variable is greater than [max] (when [incr] is positive or 0) or less than [max] (when [incr] is negative), the pointer jumps to the line after the end command; otherwise, it jumps back to the top. As for the variables defined by the for command, they take on the following properties:
- They are available to all commands within the script
- If there is a global variable outside the for loop with the same name as that variable, the for loop one will get picked if it gets referenced
- Variables only exist while the for loop is in effect: once the for loop ends, the variables are no longer available (unless they were created earlier, in which case their values are restored)
- Nested for loops have access to all variables from the previous for loops
- None of the numbers in the command can be clauses; [init], [max], and [incr] must all be single integers or variables with single integer values
- jump and call commands can access labels outside of the for loop, but you should make sure you return to where it was called. Each end command is paired to a specific for command. If you have nested for loops and the pointer reaches the wrong end command because of jumping, the loops may function incorrectly.

 Note that variables created with create commands within the loop are global, and will not disappear.

 Here is an example of three for loops that do the same thing: 
	for	x	100	200	10
		left	x	100
		pause	1000
	end
	
	create	int	total	0
	for	x	0	-100	-10
		set	total	-	100	x
		left	total	100
		pause	1000
	end
	
	for	x	0	-100	-10
		left	-	100	x	100
		pause	1000
	end
 These will left click at x-100, y-100, pause a second, left click at x-110, y-100, pause a second, and so on. At the end, they will click at x-200, y-100, pause a second, and then continue on. A similar set of for loops is available in example/019.

	input:

		[init] - starting variable value
		[max] - maximum variable value
		[incr] - increment to use each loop
	output:
		- when reached for the first time:
			- creates a variable with starting value [init]
		- when jumped to by an end command:
			- adds [incr] to [name]
			- checks if either ([incr] is positive or 0 and [name] is greater than [max]) or
				([incr] is negative and [name] is less than [max])
			- if the check is false, proceeds to the next command
			- if the check is true, jumps to the command after the end command

end:

 Run also provides some miscellaneous options that can be set with these two commands: 
	optionon	[setting]
	optionoff	[setting]
 This will either turn on or turn off, respectively, certain settings that can be helpful while running scripts. All settings apply only to the script that called it. optionon settings may take extra parameters, but optionoff only needs the name of the option. There are currently seven available settings.
 The first one, 
	optionon	quitafter	[number]
takes an integer parameter, [number]. Since pauses and pausenots only continue once the pixel changes, the script might run indefinitely if the pixel doesn't change properly. This option will end the script if [number] milliseconds pass without the pixel changing. Note that this is different than an extra parameter at the end of the pause/pausenot, because that parameter will continue running the script. quitafter is off by default.
 The second option, 
	optionon	endingcall	[label]
triggers when the script ends in the middle "abnormally", which happens when the mouse is moved during a pause or pausenot, a quit command is called, or a pause reaches its quitafter point. endingcall jumps to the label [label], and turns off the endingcall option. It can be reenabled if there is another optionon endingcall command after the label, but when the script ends it will always end "abnormally". endingcall is not called for scripts that end "normally" (by reaching the bottom of the script). If a script ends "abnormally", then any scripts that called it will end abnormally as well, triggering those scripts' endingcalls if they were set. You can, however, have future run commands which will not automatically end "abnormally". endingcall is off by default.
 The third option, 
	optionon	variance	[red]	[green]	[blue]
is for use with checking pixel colors. The variance is how much higher or lower each of the three color values (red, green, blue) can be to be accepted as correct (this is why pauses only say "approximately" the right color). If a value of the variance is 5, then in any command that uses pixel colors, an acceptable pixel value can be between 5 higher and 5 lower than the given value. If the red variance is 5, checking for a red of 128 would accept anything from 123 to 133 as the right color value. You can also use 
	optionon	variance	[value]
to set [value] to all three color variances. variance is 5 5 5 by default. Turning off the variance sets it to 0 0 0. Setting a negative variance is a bad idea.
 The fourth option, 
	optionon	mousetolerance	[xtolerance]	[ytolerance]
allows you to adjust the how easy it is to end the script "abnormally" in a pause command. Setting the tolerance will set how much distance the mouse can move during a pause. When a pause command executes, it records the mouse position at the beginning. If the mouse moves too far away from that spot, the script will end "abnormally". If the tolerance were 0 0, then any mouse movement would trigger an "abnormal" ending. If the tolerance were 5 10 and the mouse started at x-115, y-130, then the mouse could move anywhere in the rectangle within x-110-120 and y-120-140 without triggering an abnormal ending. Like variance, you can also use 
	optionon	mousetolerance	[tolerance]
to set [tolerance] to both the x and y tolerance. mousetolerance is 0 0 by default. Turning off mousetolerance sets it to this default. Setting a negative mousetolerance is a bad idea.
 The fifth option, 
	optionon	scanorder	[direction]	[direction]
sets the order in which the scan clause runs. The first direction is the primary direction in which the scan searches the picture, the second direction is the secondary direction. The directions for the command are up, down, left, and right. If you had the following, 
	optionon	scanorder	right	up
then scans would start at the bottom left corner, go right along the bottom row of pixels, and once the end has been reached, it would move up one row of pixels, go the left side, and start again. If your first direction is invalid (not up, down, left, or right), the scanorder does not get set. If your second direction is invalid, or is on the same axis as the first direction (vertical or horizontal), a default second direction will be assigned: down for vertical and right for horizontal. The default scanorder is right then down. Turning off scanorder sets it to this default.
 The sixth option, 
	optionon	offset	[x]	[y]
allows you to set an offset to apply to all interactions with the screen. Any command or operator that uses screen coordinates (move, pausenot, scan, etc.), including ones used with screenshots, will shift by this amount relative to the physical location of the screen, essentially setting the screen so that this new location is the top left corner. Setting an offset will override any old one. For example, if you had a script with the following commands, 
	optionon	offset	23	57
	optionon	offset	50	100
	move	25	37
the mouse would move to the physical position x-75, y-137. Similarly, 
	optionon	offset	2	43
	pause	10	10	255	255	255
would wait until the color at x-12, y-53 is the right color. All variables are unaffected, and .scanx, .scany, .mousex, and .mousey take the offset into account. If you had 
	optionon	offset	34	56
	if	scan	10	10	20	20	30	30	30
		[command]
the scan would start at x-44, y-66. If the pixel at x-44, y-66 is the correct color, then .scanx would be 10 and .scany would be 10. If you then had 
	move	.scanx	.scany
your mouse would move to x-44, y-66. Similarly, for .mousex and .mousey, if you had 
	optionon	offset	12	33
	create	int	mx	.mousex
	create	int	my	.mousey
	move	mx	my
	
	optionon	offset	12	33
	move	.mousex	.mousey
and your mouse was physically at x-125, y-150, then for the first part, mx and my would be 113 and 117, respectively, and the mouse would effectively not move. The second part would do the same thing, moving the mouse nowhere. Setting offset to 0 0 is equivalent to turning it off. offset is off by default.
 The seventh option, 
	optionon	endmessages
sets whether or not Run will display a descriptive ending message when the program exits "abnormally". endmessages is on by default.
